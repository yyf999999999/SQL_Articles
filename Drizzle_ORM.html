Drizzle ORMで学ぶSQLの実行性能

はじめに

本教材の背景

Webアプリケーション開発では、データベースを利用したデータの保存や検索が不可欠です。近年の開発現場では、SQL文を直接記述する代わりに、ORM（Object-Relational Mapping）と呼ばれる仕組みを利用してデータベースにアクセスする方法が広く使われています。ORMを利用すると、プログラミング言語のコードとしてデータベース操作を記述できるため、可読性の向上や型安全性の確保、SQLインジェクションの防止といった利点があります。

しかしその一方で、ORMはSQLを抽象化するため、実際にどのようなSQL文が発行されているのか、またそのSQLがデータベース内部でどのように実行されているのかが見えにくくなるという側面もあります。データベースの処理性能は、インデックスの有無やデータ量、クエリの構造、実行計画などに大きく影響されます。そのため、ORMを利用している場合でも、データベース内部の仕組みに関する理解は依然として重要です。

本教材の目的

本教材の目的は、ORMを利用したアプリケーション開発と、データベース内部の動作との関係を実験を通して理解することです。

本教材では、TypeScript環境で利用できる軽量なORMである Drizzle ORM を用いてPostgreSQLに接続し、データベース操作を行います。そして、ORMが発行するSQL文を確認するとともに、そのSQLがPostgreSQL上でどのような実行計画に基づいて処理されるのかを検証します。

特に、次の点に注目します。

  • ORMで記述した処理が、どのようなSQL文に変換されているか
  • インデックスの有無によって実行計画や処理時間がどのように変化するか
  • JOINを含むクエリにおいて、データベースがどのような方法でテーブルを結合しているか

これらを実験によって確認することで、「ORMを使えばデータベースのことを意識しなくてもよい」という考えが必ずしも正しくないこと、そしてORMの知識とデータベースの知識は互いに補完し合う関係にあることを理解することを目標とします。

ORMの概要とDrizzle ORMの特徴

ORM（Object-Relational Mapping）とは

ORM（Object-Relational Mapping）とは、プログラム上のオブジェクトとリレーショナルデータベースのテーブルを対応付ける仕組みのことです。

リレーショナルデータベースでは、データは「表（テーブル）」の形で管理されます。一方、アプリケーション（特にTypeScriptやJavaなどのオブジェクト指向言語）では、データは「オブジェクト」として扱われます。
ORMはこの「オブジェクトの世界」と「テーブルの世界」を橋渡しする役割を担います。

id name email
1 Sato sato@example.com

ORMを使うと、この1行のデータを次のようなオブジェクトとして扱えるようになります。

usersテーブルの1レコードに対応するオブジェクト例 
{
  id: 1,
  name: "Sato",
  email: "sato@example.com"
}

このように、SQLを直接書かなくても、プログラムのコードとしてデータベース操作が可能になるのがORMの基本的な考え方です。

ORMを利用するメリットと注意点

ORMは便利な仕組みですが、万能ではありません。ここでは利点と注意点を整理します。

■ ORMの主なメリット

  • SQLを直接書かずにデータベース操作ができる
    プログラミング言語の文法で処理を書けるため、アプリケーションコードとの統一感が生まれます。
  • 型安全性を確保しやすい
    TypeScriptなどと組み合わせることで、存在しないカラム名の指定や型の不一致といったミスをコンパイル時に発見できます。
  • SQLインジェクション対策が容易になる
    多くのORMでは、値のバインド処理が自動で安全に行われるため、セキュリティリスクを減らせます。

■ ORM利用時の注意点

  • 発行されるSQLが見えにくくなる
    ORMは内部でSQLを自動生成するため、どのようなクエリが実行されているのかを意識しないまま使ってしまうことがあります。
  • パフォーマンス問題に気付きにくい
    非効率なクエリが生成されていても、コード上はシンプルに見えるため、性能低下の原因が分かりにくくなることがあります。

そのため、ORMを利用する場合でも、SQLや実行計画、インデックスといったデータベース内部の知識は依然として重要です。

主なORMの種類と特徴

世の中には多くのORMが存在し、それぞれ設計思想が異なります。ここでは代表的なタイプを簡単に整理します。

① フル機能型ORM

例：Hibernate（Java）、Entity Framework（.NET）

  • データベース操作の多くを自動化
  • マイグレーションやキャッシュ機能なども充実
  • 学習コストや設定項目が多い

② クエリビルダ寄りのORM

例：Knex.js など

  • SQLをプログラム的に組み立てる
  • SQLに近い感覚で書ける
  • 自動マッピング機能は比較的少なめ

③ 型安全性重視の軽量ORM

例：Prisma、Drizzle ORM

  • TypeScriptとの相性が良い
  • スキーマと型の連携が強力
  • 必要以上の抽象化を行わず、SQLとの距離が近い

Drizzle ORMはこの中でも「軽量かつ型安全性を重視したORM」に分類されます。

Drizzle ORMの特徴

Drizzle ORMは、TypeScript環境で利用される比較的新しいORMで、次のような特徴があります。

SQLに近い記述スタイル

Drizzleは、従来の「完全に抽象化されたORM」とは異なり、SQLの構造をある程度意識した書き方になっています。
そのため、裏でどのようなSQLが発行されるのかをイメージしやすい設計になっています。

■ 強力な型推論

テーブル定義をTypeScriptで記述すると、その情報がそのままクエリの型チェックに利用されます。

  • 存在しないカラムを指定するとエラーになる
  • 数値カラムに文字列を入れようとするとエラーになる

といったチェックが、実行前に行われます。

■ 軽量でシンプルな構成

大規模な設定ファイルや複雑なコード生成を必要とせず、比較的シンプルに導入できます。
学習コストが低く、「まずはORMを試してみたい」という用途にも適しています。

本教材におけるDrizzle ORMの位置付け

本教材では、Drizzle ORMを「データベース内部の動作を隠しすぎないORM」 として扱います。

つまり、

  • ORMを使って便利に開発する体験
  • その裏で発行されるSQLを確認する体験
  • さらにそのSQLの実行計画をPostgreSQLで分析する体験

これらを結び付けるためのツールとしてDrizzle ORMを利用します。

第3章以降では、実際にDrizzle ORMを導入し、PostgreSQLに接続してデータ操作を行いながら、ORMとデータベース内部動作の関係を具体的に確認していきます。

Drizzle ORMの導入とPostgreSQLへの接続

前提条件と授業環境

この教材では、データベース工学の授業で使用したDocker上のPostgreSQL環境（DB-PostgreSQLリポジトリ）を利用して進めます。
すなわち、npm run db:upなどで PostgreSQLを起動し、DockerコンテナのPostgreSQLを操作対象とします。

依存パッケージのインストール

まず、プロジェクトルート(DB-PostgreSQL)でDrizzle ORMと関連パッケージをインストールします。

Drizzle ORM と関連パッケージのインストール 
npm install drizzle-orm pg dotenv
npm install -D drizzle-kit tsx @types/pg
  • drizzle-orm: Drizzle ORM本体
  • pg: PostgreSQL用Node.jsドライバ
  • dotenv: .envを読み込み環境変数化
  • drizzle-kit: Drizzle用ツール（マイグレーションなど）
  • tsx: TypeScript実行ツール
  • @types/pg: pgの型定義（TypeScript 用）

警告(deprecated)が出る場合がありますが、Drizzle ORMの基本動作には影響しません。

Drizzle ORMで接続を確認するコード

次に、Drizzle ORMを使ってPostgreSQLへの接続を確認するためのコードを作成します。

以下のコードをsrc/setupDrizzle.tsなどに記述します：

Drizzle ORM接続テスト用コード 
import dotenv from "dotenv";
dotenv.config();

import { drizzle } from "drizzle-orm/node-postgres";
import { Pool } from "pg";

console.log("DATABASE_URL =", process.env.DATABASE_URL);

// PostgreSQL 接続プールを作成
const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
});

// Drizzle ORM を初期化
export const db = drizzle(pool);

// 接続テスト用関数
async function testConnection() {
  const result = await db.execute("SELECT version()");
  console.log("Connected to PostgreSQL:", result.rows);
}

testConnection().catch(console.error);
  • dotenvを最初に呼び出し.envを読み込みます
  • Poolで接続プールを作成
  • db.execute("SELECT version()")で簡単なSQLを実行

接続確認の実行

ターミナルで次のように実行すると、PostgreSQL のバージョン情報が表示されます：

接続確認の実行コマンド 
npx tsx src/setupDrizzle.ts

成功例：

接続成功時の出力例

Connected to PostgreSQL: [  
  { version: 'PostgreSQL 17.6 (…) x86_64-pc-linux-gnu (…)' }  
]

この表示は、Drizzle ORMがPostgreSQL（Docker内）へ接続し、SQL実行まで成功していることを示します。

Windowsでの競合と注意点

この節は、npx tsx src/setupDrizzle.tsで接続がうまくいかなかった場合におけるトラブルシューティングのヒントを提供しています。
上手くいった場合は、この節は読み飛ばして構いません。なお、この方法をとっても解決しない場合は、生成AIなどを活用して対応してください。

Windows上でDrizzle ORMのPostgreSQLへの接続が上手くいかなかった場合、以下のようにローカルOS側にもPostgreSQLサービスが起動しているか確認してください。 もし同じポート(5432)で他のPostgresサービスが動いていると、Node/Drizzleはそちらに接続してしまい、誤った認証エラーになることがあります。

確認方法（サービス一覧）

  1. Windows+Rキーを押してservices.mscと入力してサービス一覧を開く
  2. サービス一覧の中でPostgreSQLを含むものを探す(例：postgresql-x64-17)
  3. もし見つかったら停止しておく
  4. 再度npx tsx src/setupDrizzle.tsを実行して接続を確認する

このように競合を解消しておくことで、Docker内のPostgreSQLへの接続が成功する可能性が高まります。

例アプリのテーブル定義とCRUD

例アプリの概要

次は、Drizzle ORMの使い方を学ぶために例アプリを作成します。例アプリと言ってもUIなどアプリ画面まで作るわけではなく、CRUDの基本的な動作を確認するのみとなっています。
今回作成する例アプリは、データベースにユーザー情報を保存・取得・更新・削除する基本的な機能を持ったシンプルなアプリケーションです。

  • Create：ユーザー情報を新規登録する
  • Read：登録済みのユーザー情報を一覧・検索する
  • Update：既存ユーザーの情報を変更する
  • Delete：ユーザー情報を削除する

Drizzle ORMを用いることで、テーブル定義やSQLの実行をTypeScriptで書けるようになり、型安全かつ簡潔にデータベース操作が記述できます。また、今回作成するアプリは最小限の機能に留めているので、Drizzle ORMやPostgreSQLの基本を学ぶための導入として最適な題材です。

テーブル定義（schema.ts）

まずは、ユーザー情報を保存するためのテーブルを定義します。Drizzle ORMでは、pgTable関数を用いてテーブル構造をTypeScriptで記述できます。

今回はテーブル名をdrizzle_usersとし、以下の3つのカラムを持たせます。

  • id：主キー（自動採番）
  • name：ユーザー名
  • email：メールアドレス
schema.ts 
import { pgTable, serial, varchar } from "drizzle-orm/pg-core";

export const drizzleUsers = pgTable("drizzle_users", {
  id: serial("id").primaryKey(),
  name: varchar("name", { length: 256 }).notNull(),
  email: varchar("email", { length: 256 }).notNull(),
});

Drizzle Kit の設定ファイルの作成 (drizzle.config.ts)

Drizzle ORMでマイグレーションを自動生成したり適用したりするためには、設定ファイルが必要です。 Drizzle Kitはdrizzle.config.tsというファイルをルートに置き、ここにスキーマの場所や使用するデータベースの種類などを指定します。

プロジェクトのルート（package.jsonと同じ階層）に以下の内容のファイルを作成してください：

drizzle.config.ts 
import 'dotenv/config';
import { defineConfig } from 'drizzle-kit';

export default defineConfig({
  schema: "./src/db/schema.ts",
  dialect: "postgresql",
  dbCredentials: {
    url: process.env.DATABASE_URL!,
  },
});
  • schema：Drizzle ORM のスキーマ定義ファイルへのパス
  • dialect：利用するデータベースの種類（今回はPostgreSQL）
  • dbCredentials.url：接続文字列（.envのDATABASE_URLを参照）

マイグレーションファイルの生成と適用

設定ファイルを作成したら、次のコマンドを順に実行することで、スキーマ定義に基づいたマイグレーションファイルが生成され、必要なテーブルがデータベースに追加されます。

テーブル作成 
npx drizzle-kit generate --name=init
npx drizzle-kit migrate
  • generate：スキーマの変更点からSQLマイグレーションを自動生成します。
  • migrate：生成されたマイグレーションをDBに適用します。

※注意：drizzle-kit pushでもマイグレーションを行えますが、既存テーブルとの整合性を保つためにはgenerate → migrateの方が安全です。

CRUD操作の実装（crud.ts）

次に、CRUD（Create/Read/Update/Delete）を実装します。 Drizzle ORMでは、条件式にeq()などの演算子関数を使用します。(PostgreSQLの = に相当) カラムに直接.eq()を呼び出す形式ではない点に注意してください。

crud.ts 
import { eq } from "drizzle-orm";
import { db as defaultDb } from "../setupDrizzle";
import { drizzleUsers } from "./schema";

export function createUser(
  name: string,
  email: string,
  opt: { db?: any } = {}
) {
  const d = opt.db ?? defaultDb;
  return d.insert(drizzleUsers).values({ name, email });
}

export function getAllUsers(opt: { db?: any } = {}) {
  const d = opt.db ?? defaultDb;
  return d.select().from(drizzleUsers);
}

export function updateUserEmail(
  id: number,
  newEmail: string,
  opt: { db?: any } = {}
) {
  const d = opt.db ?? defaultDb;
  return d
    .update(drizzleUsers)
    .set({ email: newEmail })
    .where(eq(drizzleUsers.id, id));
}

export function deleteUser(id: number, opt: { db?: any } = {}) {
  const d = opt.db ?? defaultDb;
  return d
    .delete(drizzleUsers)
    .where(eq(drizzleUsers.id, id));
}

この実装では、通常のdbだけでなく、トランザクション用のtxも渡せるように設計しています。 これにより、後述するトランザクション処理でも同じCRUD関数を利用できます。

アプリ全体の実行（app.ts）

次に、CRUD処理をまとめて実行するアプリ本体を作成します。 ここではトランザクションを開始し、最後に明示的にrollbackを実行して動作確認を行います。

app.ts 
import { db } from "../setupDrizzle";
import {
  createUser,
  getAllUsers,
  updateUserEmail,
  deleteUser,
} from "./crud";

async function main() {
  try {
    await db.transaction(async (tx) => {
      console.log("=== Transaction Start ===");

      await createUser("Tanaka Ichiro", "ichiro@example.com", { db: tx });
      console.log("After create:", await getAllUsers({ db: tx }));

      const users = await getAllUsers({ db: tx });
      console.log("Read check:", users);

      if (users.length > 0) {
        const id = users[0].id;
        await updateUserEmail(id, "taro.new@example.com", { db: tx });
      }
      console.log("After update:", await getAllUsers({ db: tx }));

      if (users.length > 0) {
        const id = users[0].id;
        await deleteUser(id, { db: tx });
      }
      console.log("After delete:", await getAllUsers({ db: tx }));

      console.log("=== Forcing Rollback ===");
      tx.rollback();
    });
  } catch (e: any) {
    if (e?.message === "Rollback") {
      console.log("Transaction was rolled back successfully.");
    } else {
      console.error("Unexpected error:", e);
    }
  }

  console.log("=== After Transaction (Outside) ===");
  console.log("Final DB state:", await getAllUsers());
}

main();

トランザクションとRollbackの確認

Drizzle ORMでは、db.transaction()内で例外が発生すると自動的にrollbackされます。 また、tx.rollback()を明示的に呼び出すことでもロールバック可能です。

ただし、PostgreSQLの仕様上、シーケンス（serial）の値はrollbackされません。 そのため、IDが途中で飛ぶように見えることがありますが、これは正常な挙動です。

  • INSERT → rollback → データは消える
  • しかし nextval()は戻らないためIDは進む

実行方法

以下のコマンドで実行できます。

アプリの実行コマンド 
npx tsx src/db/app.ts

ログに「Transaction was rolled back successfully.」と表示され、 トランザクション外で確認したデータが元の状態に戻っていれば成功です。

以上で、Drizzle ORMを用いたテーブル定義およびCRUD操作、 さらにトランザクション制御まで含めた基本的な例アプリが完成しました。 これらの構造は実際のWebアプリケーション開発でもほぼ同じ形で利用されます。

Create 操作の解説

ここではcrud.tsのcreateUser関数を例に、INSERT操作を確認します。

createUser関数

  name: string,
  email: string,
  opt: { db?: any } = {}
) {
  const d = opt.db ?? defaultDb;
  return d.insert(drizzleUsers).values({ name, email });

中でも、d.insert(drizzleUsers).values({ name, email })の部分が、Drizzle ORMでのINSERT操作の核心です。

Drizzleでは.insert()にテーブルオブジェクトを渡し、.values()にオブジェクト形式で値を指定します。

テーブル定義と型安全性

drizzleUsersはschema.tsで定義したテーブルオブジェクトです。
drizzleUsersに基づいて型が自動推論されるため、

  • 存在しないカラム名を指定するとコンパイルエラー
  • 型不一致もコンパイル時に検出

というメリットがあります。

opt.dbの設計意図

  • 通常実行 → defaultDb
  • トランザクション内 → tx

のどちらにも対応できる設計です。
つまりCRUD関数自体はトランザクションの存在を意識しない設計になっています。

Read 操作の解説

次に、crud.tsのgetAllUsers関数を例に、SELECT操作を確認します。

getAllUsers関数

  const d = opt.db ?? defaultDb;
  return d.select().from(drizzleUsers);
}

中でも、d.select().from(drizzleUsers)の部分が、Drizzle ORMでのSELECT操作の核心です。
この処理は、PostgreSQLでは次のSQLに相当します。

getAllUsers関数に対応するSQL 
SELECT * FROM drizzle_users;

Drizzleのselect()は「ビルダー方式」です。

  • .select() → SELECT 開始
  • .from(drizzleUsers) → FROM 句
  • .where() → WHERE 句
  • .orderBy() → ORDER BY 句

という流れでSQLの構造とほぼ1対1対応しています。

Delete 操作の解説

インデックスと実行計画

本章の目的

第4章ではCRUDとトランザクションを確認しました。 本章ではインデックスの有無による検索性能の違いを、Drizzle ORMのコードから直接確認します。

Drizzle ORMはTypeScriptからSQLを生成し、PostgreSQLに送信しますが、どの実行計画を使うか（Seq ScanやIndex Scanなど）の選択はPostgreSQL側が行います。

検証用データの投入コード

大量データの投入と既存データ削除を行うsrc/db/seed.tsを作成します。

src/db/seed.ts

    users.push({

実行計画をコードから出力する

インデックスなし/ありの実行計画をDrizzle ORMのコードから取得して出力するsrc/db/explainIndex.tsを作成します。

  • Create Index：インデックスを作成する
  • Explain With Index：インデックスありの検索でIndex Scanが使われる

実行計画の出力を比較することで、インデックスによって検索方法が変わり、処理効率が改善することが確認できます。

インデックス操作と走査方法の原理

PostgreSQLではテーブルに値を検索する際、いくつかの探索方法があります。 ここではインデックスなし・ありの場合の基本的な仕組みを説明します。

Seq Scanはテーブル全体を先頭から最後まで順番に読み出して条件に合う行を探す方法です。 この方式はインデックスがない場合やテーブルが小さい場合に利用されますが、テーブルサイズが大きい場合に読み取り量が多くなるため時間がかかることがあります。

Index Scanは先にインデックスをたどって、条件に該当する行の位置を高速に絞り込みます。 インデックスを使うことで必要な行だけを効率的に読み出すことができるため、大量データの検索が高速になります。 ただしインデックスの更新や作成には追加のコストがかかります。

確認テスト

問1 インデックスなしの実行計画はSeq Scanです。

問2 インデックスありの実行計画はIndex Scanです。

問3 インデックスは検索を高速化します。

ここでは、複数テーブルを結合するクエリ（JOIN）をDrizzle ORMで書き、 実際に発行されるSQLをPostgreSQLで実行して実行計画の違いを比較します。

schema.ts の最終版ファイル

この章で使用するJOIN用テーブル定義を含め、完全版のsrc/db/schema.tsを以下に示します。

JOINにおける実行計画のポイント

JOINクエリでは複数のテーブルが結合されるため、結合キーのインデックスが性能に強く影響します。
以下に代表的な違いをまとめます。

  • インデックスなし：PostgreSQLは結合条件に対してSeq Scanを使い、全件を読みながら結合結果を探します。
  • インデックスあり：インデックスを使って一致する行を効率的に絞り込み、結合処理を高速化できます。

実行計画（簡単な解説）

PostgreSQLはEXPLAIN ANALYZEの結果から、どの方式で結合を行うかを決定します。
主に以下のような探索方法があります：

  • Seq Scan：テーブル全体を順番に読み出す方式
  • Index Scan：インデックスを使って一致する行を効率的に読み出す方式
  • Bitmap Index Scan：複数条件を効率的に扱う方式

この違いがExplainの結果として現れ、性能差として実行時間にも反映されます。

確認テスト

問1 PostgreSQLでは、インデックス検索で該当する行がある程度多い場合、複数の行位置をまとめて取得するBitmap Index Scanが選択されることがあります。

問2 次のうち、JOINクエリで結合順序や結合方法が実行計画に影響する要素として正しいものはどれですか？
A. 結合するテーブルの行数だけ
B. 結合キーにあるインデックスの有無
C. 関連しないテーブルを含むかどうか
D. 実行前のSQLコメントの有無

答：B

集約処理（GROUP BY）と実行計画

この章の概要

第6章では複数テーブルの結合（JOIN）とその実行計画を確認しました。 本章では、Webアプリでよく使われる集約処理（GROUP BY）について学びます。 さらに、PostgreSQLが集約処理をどのように実行するかを実行計画（EXPLAIN ANALYZE）で確認します。

実装内容

本章では以下の処理を行います：

  • 各グループに所属するユーザー数を集計する
  • GROUP BYを用いた集計結果を取得する
  • 同じ処理の実行計画を確認する

使用するテーブルは第6章のまま（drizzle_users、 drizzle_groups、 drizzle_user_groups）です。

集約処理の実装（explainAggregate.ts）

以下のプログラムsrc/db/explainAggregate.tsでは、GROUP BYによる集計結果の取得と 同じ処理の実行計画を確認します。

src/db/explainAggregate.ts 
import "dotenv/config";
import { db } from "../setupDrizzle";
import { drizzleGroups, drizzleUserGroups } from "./schema";
import { sql } from "drizzle-orm";

async function main() {
  console.log("DATABASE_URL =", process.env.DATABASE_URL);

  // PostgreSQL接続確認
  const version = await db.execute(sql`SELECT version() AS version`);
  console.log("Connected to PostgreSQL:", version.rows);

  console.log("=== Aggregate Result ===");

  // 集計結果の取得（生SQL）
  const result = await db.execute(sql`
    SELECT
      g.group_name,
      COUNT(ug.user_id) AS user_count
    FROM drizzle_groups g
    LEFT JOIN drizzle_user_groups ug
      ON g.id = ug.group_id
    GROUP BY g.id, g.group_name
    ORDER BY g.id;
  `);

  console.log(result.rows);

  console.log("=== Explain Aggregate ===");

  // 実行計画の確認
  const explain = await db.execute(sql`
    EXPLAIN ANALYZE
    SELECT
      g.group_name,
      COUNT(ug.user_id)
    FROM drizzle_groups g
    LEFT JOIN drizzle_user_groups ug
      ON g.id = ug.group_id
    GROUP BY g.id, g.group_name;
  `);

  console.log(explain.rows);
}

main().catch(console.error);

実行方法

以下のコマンドを実行すると、集計結果とその実行計画が表示されます：

実行コマンド 
npx tsx src/db/explainAggregate.ts

実行計画の読み取り

実行計画には以下のような行が表示されることがあります：

実行計画の例(抜粋) 
HashAggregate
Group Key: g.id

HashAggregateは、PostgreSQLがGROUP BY処理を効率的に行うために 使用する最適化手法です。Hashテーブルを使ってグループごとに集計します。

この実行計画では、以下の流れで処理されています：

  • テーブルを全件読み込む（Seq Scan）
  • 結合処理を行う（Hash Join）
  • 集計処理を行う（HashAggregate）

PostgreSQLはデータ量や統計情報に基づいて、最適な処理方法を自動で選択します。
